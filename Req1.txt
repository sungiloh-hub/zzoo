1. 프로젝트 개요 – 기획 배경 및 핵심 내용
================================================================================
 
1.1 프로젝트 기획 배경
 
•  어떤 문제를 해결하고자 하는가?
   직장인은 하루 권장 칼로리를 초과 섭취하는 경우가 많으며, 구내식당 중식 이후
   저녁에 얼마나 먹어야 하는지를 직접 계산하기 어렵다. 칼로리 관리 없이 저녁을
   선택함으로써 비만, 대사질환 등 건강 문제가 누적되는 것을 해결하고자 한다.
 
•  기존 방식의 한계는 무엇인가?
   - 일반 칼로리 계산 앱: 매끼 음식을 직접 검색·입력해야 하며 구내식당 메뉴와 연동 불가
   - 식단 추천 서비스: 개인의 당일 섭취 현황을 반영하지 않고 일반적인 추천만 제공
   - 구내식당 메뉴표(PDF): 단순 열람만 가능하며 칼로리 정보 및 추천 기능 없음
 
•  Agent 서비스로 해결할 수 있는 Pain Point는 무엇인가?
   - 오늘 날짜를 자동으로 감지하여 별도 날짜 입력 없이 당일 메뉴 즉시 조회
   - 구내식당 4개 코스(윈디쉬·진국·면가·샐러데이) 중 사용자가 선택한 코스의
     칼로리를 자동으로 표시
   - 목표 칼로리 - 선택한 중식 칼로리 = 저녁 잔여 칼로리 자동 산출
   - 잔여 칼로리와 영양 균형을 고려한 개인화된 저녁 메뉴를 자연어로 추천
   - 사용자는 목표 칼로리 입력 + 중식 코스 선택만 하면 나머지를 Agent가 처리
 
•  이 프로젝트를 시작하게 된 동기는 무엇인가?
   바쁜 직장인이 별도의 노력 없이 건강한 식습관을 유지할 수 있도록, 실제로 이용하는
   구내식당 4개 코스 메뉴와 개인 칼로리 목표를 연계하는 스마트 저녁 메뉴 추천
   서비스의 필요성을 직접 느끼고 기획하게 되었다.

1.2 핵심 아이디어 및 가치 제안(Value Proposition)
 
•  서비스가 제공하는 핵심 기능은 무엇인가?
   - 하루 목표 칼로리 입력 (날짜는 오늘 날짜 자동 적용)
   - 구내식당 4개 중식 코스 카드 표시 및 선택
     ① 윈디쉬 (Western Dish)
     ② 진국 (국·탕류)
     ③ 면가 (면류)
     ④ 샐러데이 (샐러드류)
   - 선택한 코스의 오늘 메뉴 구성 + 칼로리 자동 표시 (RAG 기반)
   - 잔여 칼로리(목표 - 선택 코스 칼로리) 자동 계산
   - 영양소(탄수화물·단백질·지방) 균형을 고려한 저녁 메뉴 3종 추천
   - 추천 이유, 영양소 정보, 대체 메뉴 옵션 제공
 
•  사용자에게 제공되는 가치와 기대효과는 무엇인가?
   - 간편한 관리: 목표 칼로리 입력 + 코스 선택만으로 즉시 저녁 메뉴 추천
   - 건강 증진: 영양 균형 잡힌 저녁 식단으로 체중·건강 관리 효과
   - 시간 절약: 날짜 입력·메뉴 검색·칼로리 계산에 소요되는 시간 대폭 단축
   - 개인화: 당일 선택한 중식 코스 기반 실시간 맞춤형 저녁 메뉴 추천
 
•  기존 서비스 대비 차별성은 무엇인가?
   - 오늘 날짜 자동 감지로 입력 단계 최소화 (목표 칼로리 + 코스 선택 2단계만)
   - 구내식당 4개 코스를 카드 UI로 선택하는 직관적 UX
   - 선택 즉시 해당 코스 칼로리 표시 → 잔여 칼로리 기반 저녁 추천까지 원스톱 제공
   - LLM 기반 자연어 추천으로 단순 수치 나열이 아닌 이유와 함께 메뉴 제안

1.3 대상 사용자 및 기대 사용자 경험(UX)
 
•  주요 타겟
   - 1차 타겟: 해당 구내식당(윈디쉬·진국·면가·샐러데이 운영) 이용 직장인
   - 2차 타겟: 체중 관리 중인 직원, 당뇨·고혈압 등 식이 관리가 필요한 임직원
 
•  사용자에게 어떤 흐름과 경험을 제공할 것인가?
   ① 웹앱 접속 → 오늘 날짜 자동 표시 + 하루 목표 칼로리 입력 (예: 2,000kcal)
   ② 오늘의 4개 중식 코스 카드 자동 표시
      [윈디쉬] [진국] [면가] [샐러데이]
   ③ 사용자가 오늘 먹은(또는 먹을) 코스 1개 선택
   ④ 선택한 코스의 오늘 메뉴 구성 및 칼로리 자동 표시
      예) 윈디쉬 선택 → "오늘 윈디쉬: 치킨스테이크 정식 (720kcal)"
   ⑤ 잔여 칼로리 자동 계산 및 저녁 메뉴 추천 카드 3종 표시
   ⑥ 추천 이유, 영양소 정보, 대체 메뉴 옵션 확인 및 선택
 
•  사용자가 서비스에서 얻는 구체적 Benefit은 무엇인가?
   - 매일 1분 이내에 오늘 저녁 메뉴 결정 가능
   - 날짜 입력 없이 오늘 메뉴 즉시 확인
   - 칼로리·영양 지식이 없어도 건강한 식단 유지
   - 4개 코스 중 실제 선택한 메뉴 기반의 현실적 저녁 추천 제공

2. 기술 구성 – 서비스에 적용할 기술 스택

2.1 Prompt Engineering 전략
 
•  역할 기반 프롬프트
   시스템 프롬프트에 전문가 역할 명시:
   "당신은 영양 전문가이자 식단 코치입니다. 사용자가 오늘 구내식당에서 선택한
   중식 코스(윈디쉬·진국·면가·샐러데이)의 칼로리를 기반으로 잔여 칼로리와
   영양소 균형을 고려하여 현실적인 저녁 메뉴를 추천합니다."
 
•  CoT/Few-shot 등 고품질 응답 전략
   - Chain-of-Thought(CoT): 선택 코스 칼로리 확인 → 잔여 칼로리 계산 →
     영양소 부족분 파악 → 메뉴 후보 선정 → 최종 추천 순으로 단계별 추론 유도
   - Few-shot: 코스별 예시 입출력 쌍 포함으로 일관된 추천 포맷 학습 유도
     예) 입력: "목표 2000kcal, 윈디쉬(720kcal) 선택" →
         출력: "잔여 1280kcal / 추천: 닭가슴살 샐러드(350kcal), 된장찌개 정식(480kcal)..."
     예) 입력: "목표 1800kcal, 샐러데이(320kcal) 선택" →
         출력: "잔여 1480kcal / 추천: 단백질 위주 한식 정식(600kcal)..."
 
•  출력 구조화 템플릿 정의
   JSON 형태의 구조화된 응답 반환:
   {
     "today_date": "2025-03-23",
     "selected_course": "윈디쉬",
     "lunch_menu": "치킨스테이크 정식",
     "lunch_calories": 720,
     "remaining_calories": 1280,
     "recommendations": [
       {
         "menu_name": "메뉴명",
         "calories": 350,
         "protein": 30,
         "carbs": 25,
         "fat": 8,
         "reason": "추천 이유",
         "alternatives": ["대체메뉴1", "대체메뉴2"]
       }
     ]
   }
 
•  사용자 유형/상황별 프롬프트 분기
   - 샐러데이 선택 (저칼로리) → 잔여 칼로리 많음 → 단백질·복합탄수화물 강조 추천
   - 윈디쉬·진국 선택 (중~고칼로리) → 균형 잡힌 일반 저녁 식단 추천
   - 면가 선택 (탄수화물 높음) → 저탄수·고단백 저녁 메뉴 위주 추천
   - 잔여 칼로리 < 300kcal → 저칼로리·고단백 위주 추천 (샐러드, 두부, 나물 등)

2.2 LangChain / LangGraph 기반 Agent 구조
 
•  Multi-Agent 설계 개념
   LangGraph를 사용하여 각 역할을 담당하는 Agent를 노드(Node)로 구성하고,
   Orchestrator가 전체 흐름을 조율하는 Directed Graph 구조로 설계
 
•  각 Agent의 역할(Role) 정의
   - Date Agent           : 오늘 날짜 자동 감지 및 시스템에 전달
   - Menu Loader Agent    : PDF RAG에서 오늘 날짜 기준 4개 코스 전체 메뉴 로드
   - Course Selector Agent: 사용자가 선택한 코스(윈디쉬·진국·면가·샐러데이)의
                            오늘 메뉴 구성 및 칼로리 반환
   - Calorie Calculator Agent : 목표 칼로리 - 선택 코스 칼로리 = 잔여 칼로리 산출
   - Nutrition Analyzer Agent : 영양소(탄·단·지) 균형 분석 및 부족 영양소 파악
   - Recommendation Agent : 잔여 칼로리 + 영양 분석 기반 저녁 메뉴 최종 추천 생성
   - Orchestrator Agent   : 전체 흐름 조율 및 각 Agent 간 State 전달 관리
 
•  Tool Calling, ReAct, Memory 활용 여부
   - Tool Calling: PDF 파싱 Tool / 날짜 자동 조회 Tool / 영양 정보 API Tool
   - ReAct 패턴: 추론(Reasoning) → 행동(Action) → 관찰(Observation) 반복으로
     메뉴 파싱 오류 발생 시 재시도 로직 구현
   - Memory: 세션 기반 단기 메모리로 당일 선택 코스, 목표 칼로리, 잔여 칼로리 유지
 
 
--------------------------------------------------------------------------------
2.3 문서 전처리 및 데이터 파이프라인 (RAG 대체)
 
•  데이터 수집/전처리 파이프라인
   ① 구내식당 주간 메뉴 PDF 준비
   ② PyPDF2 등 라이브러리로 PDF 텍스트 전체 추출
   ③ LLM (Gemini 등)의 Structured Output을 활용하여 전체 텍스트에서 날짜별, 코스별(윈디쉬·진국·면가·샐러데이) 메뉴 및 예상 칼로리를 정확하게 JSON 형태로 파싱
   ④ 처리 시간 단축 및 API 비용 절감을 위해 파싱 구조체를 로컬 DB(JSON 캐시)에 영구 저장
 
•  데이터 저장소 및 검색 구조
   - 파싱 엔진: LLM 기반 의미론적 구조화 추출 (Structured Output)
   - 데이터 저장소: 로컬 JSON 파일 (menu_db.json) 기반 초고속 로드 (Vector DB 대체)
   - 캐싱 로직: 오늘 날짜 기준으로 JSON DB를 먼저 확인하고, 식단이 있으면 즉시 반환, 없으면 최초 1회만 PDF를 읽고 LLM으로 파싱하여 캐시에 저장
 
•  검색 로직과 응답 생성 방식
   - 오늘 날짜 자동 감지 → 로컬 DB 캐시를 통해 4개 코스 전체 메뉴 일괄 로드
   - 코스 선택 시 즉시 로드된 메뉴 정보 중 해당 코스의 명칭 및 칼로리 반환
   - 반환된 코스 칼로리 + 잔여 칼로리를 추천 Agent의 프롬프트 컨텍스트에 주입

2.4 서비스 개발 및 패키징 계획
 
•  UI/UX 개발 원칙 (Clean & Minimalist)
   - Frontend: Next.js 14 (App Router) + Tailwind CSS + shadcn/ui
   - 디자인 컨셉: 'Minimalist & Intuitive' (불필요한 장식 배제, 직관적인 플랫 디자인 & 카드형 UI)
   - 레이아웃 구성:
     [상단] 오늘 날짜 자동 표출 및 심플한 목표 칼로리 입력 폼 (여백을 살린 큰 폰트)
     [중단] 4개 구내식당 코스 선택 카드 (텍스트를 최소화하고 식별성 높은 컬러 아이콘 적용)
          [ 윈디쉬 ]  [ 진국 ]  [ 면가 ]  [ 샐러데이 ]
          → 선택 시 부드러운 애니메이션과 함께 오늘 메뉴 및 칼로리 팝업/확장 표시
     [하단] 직관적인 원형/바(Bar) 형태의 잔여 칼로리 표시 + 맞춤형 저녁 추천 메뉴 카드 3종
   - 사용자 경험(UX): 모바일 최적화 반응형 디자인, 페이지 이동 없는 One-Page 스크롤 뷰 구현
 
•  BE(API) 및 배포 전략
   - Backend: FastAPI (Python 3.11+) + LangChain + LangGraph
   - API 엔드포인트:
     GET  /api/menu/today          → 오늘 4개 코스 전체 메뉴 + 칼로리 반환
     POST /api/recommend           → 선택 코스 + 목표 칼로리 수신 → 저녁 추천 반환
   - 컨테이너화: Docker + Docker Compose로 FastAPI + ChromaDB 통합 운영
   - 배포: Frontend → Vercel / Backend → Railway 또는 GCP Cloud Run
 
•  설정/환경 관리 계획
   - .env 파일로 API 키(OpenAI, Anthropic) 및 DB 경로 관리
   - python-dotenv + pydantic Settings로 환경 변수 검증
   - GitHub Actions CI/CD 파이프라인으로 자동 테스트 및 배포

2.5 선택적 확장 기능
 
•  LLM Fundamentals 기반 Structured Output / Function Calling
   - Pydantic 모델 기반 JSON 응답 스키마 강제로 안정적인 구조화 출력 보장

   - 신규 구내식당 메뉴 PDF 업로드 시 자동으로 파싱·임베딩 파이프라인 트리거
 
•  A2A 기반 Agent 협업 구조
   - Nutrition Analyzer Agent ↔ Recommendation Agent 비동기 협업 구조
   - Agent-to-Agent(A2A) 프로토콜로 분석 결과를 추천 Agent에 자동 전달
   - 향후 운동량 연동 Agent 추가 시 확장 가능한 모듈형 설계

3.1 사용자 시나리오(Use Case Scenario)
 
•  사용자 목표와 과제 흐름
   목표: 오늘 구내식당에서 선택한 중식 코스를 입력하면, 하루 목표 칼로리에 맞는
         건강한 저녁 메뉴를 자동으로 추천받는다
   과제: 목표 칼로리 입력 + 중식 코스 선택(2단계)만 하면 AI가 나머지를 처리
 
•  서비스 이용 단계별 행동 정의
 
   [Step 1] 접속 및 목표 칼로리 입력
   - 사용자가 웹앱(Next.js)에 접속
   - 오늘 날짜 자동 표시 (별도 입력 불필요)
   - 하루 목표 칼로리 입력 (예: 2,000kcal)
 
   [Step 2] 오늘 중식 코스 선택
   - 오늘의 4개 중식 코스 카드 자동 표시:
     [윈디쉬] [진국] [면가] [샐러데이]
   - 사용자가 오늘 먹은 코스 1개 선택
   - 선택 즉시 해당 코스의 오늘 메뉴 구성 + 칼로리 자동 표시
     예) [윈디쉬 선택] → "치킨스테이크 + 미역국 + 반찬 3종 / 총 720kcal"
 
   [Step 3] 잔여 칼로리 자동 계산
   - Calorie Calculator Agent: 2,000 - 720 = 잔여 1,280kcal 자동 산출
   - Nutrition Analyzer Agent: 선택 코스의 탄·단·지 분석 → 부족 영양소 파악
     예) 윈디쉬 선택 시 탄수화물 과다 → 저녁은 단백질·채소 위주 추천 유도
 
   [Step 4] 저녁 메뉴 추천
   - Recommendation Agent: 잔여 칼로리 + 영양 분석 + LLM으로 저녁 메뉴 3종 생성
   - 각 메뉴별 칼로리, 영양소, 추천 이유, 대체 옵션 포함
 
   [Step 5] 결과 확인
   - Next.js UI에 추천 메뉴 카드 형태로 표시
   - 사용자가 메뉴 선택 또는 대체 옵션 탐색

4. UI/UX 화면 설계 (Clean & Simple)
================================================================================
 
4.1 화면 주요 구성 (One-Page Application)
   사용자가 페이지를 이탈할 필요 없이, 위에서 아래로 흐르는 자연스러운 경험(Seamless Experience)을 제공합니다.
 
   [Step 1: Header & Input]
   - 여백(White Space)을 넉넉히 주어 답답함을 없앤 클린 헤더.
   - 은은한 톤의 "오늘 날짜" 위젯.
   - 중앙에 큼직하고 입력이 편한 "목표 칼로리" 입력란 구현 (경계선 없는 필드 디자인).
 
   [Step 2: Course Selection]
   - 4개의 코스 버튼을 2x2 또는 1x4 형태의 플랫 카드(Flat Card)로 단순화.
   - 복잡한 정보를 덜어내고, [윈디쉬] [진국] 등 핵심 식당 이름만 명확히 노출.
   - 클릭 시 아코디언 형태로 펼쳐지며 '오늘의 메뉴명'과 '칼로리 정보'가 말풍선처럼 깔끔하게 등장.
 
   [Step 3: Calories & Recommendation]
   - 남은 칼로리를 프로그레스 바(Progress Bar)나 도넛 차트로 심플하고 시각적으로 표현하여 현 상태 즉시 인지 가능.
   - 저녁 추천 메뉴 3종은 깔끔한 카드 형태로 수평 스와이프(Carousel) 방식으로 노출.
   - 각 추천 카드에는 [메뉴명], [태그(예: 고단백)], [필수 영양 정보]만 심플하게 기재하여 시각적 피로도 최소화.

5. 예외 상황 처리 및 에러 대응 (Robustness)
================================================================================
 
5.1 RAG 및 시스템 예외
   - 오늘의 메뉴를 불러오지 못할 경우: "메뉴 정보를 준비 중입니다"라는 심플한 스낵바(Snackbar) 알림 후, 일반 메뉴 기준으로 추천 옵션 제공.
 
5.2 사용자 입력 및 로직 예외
   - 잔여 칼로리 마이너스(초과섭취) 시: "오늘 이미 든든하게 드셨네요! 가벼운 샐러드나 걷기 등 추가 활동을 권장합니다" 긍정적인 피드백으로 유도.
   - 잘못된 칼로리 입력(예: 지나치게 큰 숫자): 실시간 유효성 검사를 통해 입력 필드 하단에 튀지 않는 색상으로 안내 문구 표시.
